# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm start          # Run the interrogator once (tsx, no build step)
npm run dev        # Watch mode — re-runs on file changes
```

No linter or test suite. TypeScript is checked implicitly by tsx at runtime.

## Environment

Copy `.env.example` (or create `.env`) in the project root:

```
OPENAI_API_KEY=
ANTHROPIC_API_KEY=
GOOGLE_API_KEY=
OLLAMA_BASE_URL=http://localhost:11434   # default; override if Ollama is remote
WEBAPP_DATA_PATH=../llmeter/src/lib/server    # where questions.json is written
WEBAPP_IMAGES_PATH=../llmeter/static/images/models  # where logos land
```

A provider is silently skipped if its API key is absent. Ollama is always attempted (key not required).

## Architecture

### Data flow

```
data/models.yml  ──┐
                   ├──▶ dataset.ts ──▶ evaluate.ts ──▶ questions.json (webapp)
data/questions.yml ┘                       │
                                           └──▶ logs/results_TIMESTAMP.log
```

1. `loadYamlModels` / `loadYamlQuestions` read from `data/`
2. `ensureLogos` checks `webappImagesPath` for existing logo files; only downloads if none found for that provider slug
3. `loadDataSet` reads the existing `questions.json` so previous responses are preserved
4. `findGaps` identifies which (question, model) pairs have no response yet
5. `runEvaluations` calls each provider in parallel (providers run concurrently; each provider's calls are sequential to respect rate limits)
6. Results are written to `questions.json` via `CoalescingWriter` as they arrive

### Question IDs

IDs are generated by `slugify(questionText, { lower: true, strict: true })`. They must be **stable** — changing question text creates a new ID and orphans old responses. Never rename a question; instead, add a new one.

### Gap-filling order (`findGaps`)

Breadth-first across providers and questions: gets one answer from each provider's first model across all questions before returning to get a second answer from the same provider. This ensures diverse coverage even if the run is interrupted.

### Write coalescing (`CoalescingWriter` in `evaluate.ts`)

`fs.writeFileSync` is synchronous but multiple providers can resolve concurrently. The writer uses a `writing` flag + `dirty` flag + `setImmediate` yield so all results that arrive while a write is in progress are batched into at most one follow-up write. Calling `drain()` after `Promise.all` ensures the final in-flight write completes before exit.

### Circuit breaker

Each provider tracks `consecutiveFailures`. After 3 in a row, remaining tasks for that provider are skipped for the session. The counter resets on any success.

### Provider quirks

- **OpenAI**: uses `max_completion_tokens` (not `max_tokens`); `temperature` is omitted entirely (gpt-5.5+ only accepts the default)
- **Google (Gemini)**: `gemini-2.5-pro` has a known intermittent bug returning empty content with `finishReason: STOP`. The provider retries up to 4 times with linear backoff before failing. `responseSchema` is passed alongside `responseMimeType` for proper structured output.
- **Ollama**: no API key; model must be pulled locally before running

### Response parsing (`providers/shared.ts`)

`parseResponse` attempts: direct JSON parse → regex extraction of `{...}` → fuzzy option matching → letter-based fallback (A/B/C). A question is only counted as failed if none of these recover a valid selection.
